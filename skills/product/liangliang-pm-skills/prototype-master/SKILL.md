# 原型大师 Prototype Master

根据PRD文档生成交互式网页原型Demo，支持数据模拟、完整交互逻辑，可直接在浏览器中运行验证。

## 核心能力

1. **PRD解析** - 理解需求文档中的功能点、交互逻辑、业务规则
2. **原型生成** - 输出单HTML文件，包含完整CSS/JS，无需后端
3. **数据模拟** - 内置模拟数据，支持增删改查操作
4. **交互实现** - 按钮点击、弹窗、表单提交、状态流转等完整交互

## 输出规范

### Demo技术要求
- **单HTML文件**：所有CSS、JS内嵌，无外部依赖
- **响应式设计**：适配桌面端，建议宽度1200px以上
- **现代浏览器**：Chrome/Firefox/Safari最新版兼容
- **数据持久化**：使用localStorage或内存状态管理

### Demo内容规范
1. **需求背景区**：展示PRD中的需求背景、痛点、解决思路
2. **功能操作区**：核心功能的交互界面（表单、按钮、表格等）
3. **数据展示区**：模拟数据展示，支持增删改查
4. **状态提示**：操作反馈（Toast提示、加载状态等）
5. **帮助说明**：操作指引、快捷键提示

### 视觉风格
- **设计语言**：现代简洁，参考Ant Design/Element Plus组件风格
- **配色方案**：主色#667eea（紫蓝渐变），成功#52c41a，警告#faad14，错误#ff4d4f
- **字体**：系统默认字体栈，无需加载外部字体
- **圆角**：统一8px圆角，按钮6px

## 工作流程

### Step 1: 分析PRD
读取用户提供的PRD文档，提取：
- 功能清单（增删改查、审批流、筛选排序等）
- 页面布局（表单、表格、弹窗、侧边栏等）
- 交互逻辑（点击事件、状态流转、数据校验等）
- 业务规则（权限控制、数据关联、计算公式等）

### Step 2: 设计Demo结构
根据PRD复杂度选择Demo类型：

| PRD类型 | Demo结构 | 示例 |
|---------|----------|------|
| 单功能配置 | 单页面 + 弹窗 | 费用标准设置、审批流配置 |
| 列表+详情 | 列表页 + 详情抽屉/弹窗 | 报销单列表、发票管理 |
| 多步骤流程 | 步骤条 + 分步表单 | 报销提单、事前申请 |
| 复杂交互 | 多标签页 + 状态流转 | 预算管理、报表系统 |

### Step 3: 生成Demo代码
基于 `assets/boilerplate.html` 模板生成完整Demo：
1. 复制模板到输出目录
2. 替换页面标题、需求背景内容
3. 实现核心交互组件
4. 添加模拟数据和状态管理
5. 测试交互流程完整性

### Step 4: 输出交付物
- **HTML文件**：可直接浏览器打开的完整Demo
- **访问方式**：
  - 方式1：下载HTML文件本地打开
  - 方式2：提供data:text/html;base64链接，直接粘贴浏览器地址栏
  - 方式3：如需在线链接，告知用户启动本地服务器

## 代码模板结构

使用 `assets/boilerplate.html` 作为基础模板：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{PRD标题}</title>
  <style>
    /* 基础样式重置 */
    /* 布局样式 */
    /* 组件样式：按钮、表单、表格、弹窗 */
    /* 状态样式：成功、警告、错误 */
  </style>
</head>
<body>
  <!-- 头部：标题+需求背景 -->
  <header>...</header>
  
  <!-- 主内容区：功能操作+数据展示 -->
  <main>...</main>
  
  <!-- 弹窗/抽屉：详情编辑、配置等 -->
  <div class="modal">...</div>
  
  <!-- Toast提示 -->
  <div class="toast">...</div>
  
  <script>
    // 状态管理
    // 数据操作函数
    // 渲染函数
    // 事件绑定
  </script>
</body>
</html>
```

## 注意事项

1. **数据模拟**：使用硬编码的模拟数据，避免需要后端接口
2. **交互完整**：确保主要交互链路可闭环（如：新增→保存→展示→编辑→删除）
3. **边界处理**：空状态、加载状态、错误提示都要有
4. **代码压缩**：精简代码，确保data URI方式可访问（控制总体大小<50KB）
5. **中文支持**：所有界面文本使用中文，符合国内产品习惯
